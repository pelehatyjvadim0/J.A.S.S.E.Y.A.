# DTO-контракты этапа 01 (`start_day` / `end_day`)

## Зачем этот документ

Этот файл фиксирует минимальные и достаточные DTO-контракты для этапа 01, чтобы:
- не использовать неявные `dict[str, Any]` между слоями;
- одинаково трактовать входы/выходы use-case;
- покрыть fallback-сценарии из плана без лишних абстракций.

Шкала необходимости поля на текущем этапе:
- `Критично` — без поля ломается основной сценарий или проверяемость контракта.
- `Желательно` — сильно упрощает эксплуатацию, диагностику и стабильность.
- `Опционально` — можно отложить до следующих этапов без блокировки релиза этапа 01.

---

## 1) Команды (`app/domain/day/commands.py`)

### `StartDayCommand`

DTO входа для запуска рабочего дня.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `user_id` | `str` | Критично | Идентификатор пользователя/профиля для чтения состояния и персонализации запуска. |
| `day_id` | `str` | Критично | Идентификатор дня в формате `YYYY-MM-DD`, нужен для трассировки и связки артефактов дня. |
| `fail_fast` | `bool` | Желательно | Режим оркестрации шагов: останавливать pipeline сразу при ошибке или продолжать с деградацией. |
| `timezone` | `str \| None` | Опционально | Часовой пояс для корректных дат и человекочитаемых отчетов. |

### `EndDayCommand`

DTO входа для завершения рабочего дня.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `user_id` | `str` | Критично | Нужен для сохранения checkpoint в правильный контекст пользователя. |
| `day_id` | `str` | Критично | Идентификатор закрываемого дня; связывает итоги с конкретным циклом. |
| `first_step_tomorrow` | `str` | Критично | Главный артефакт `end_day`: первый конкретный шаг на завтра. |
| `fail_fast` | `bool` | Опционально | Управление поведением при частичных сбоях в сборке/сохранении итогов. |

## 2) Сущности (`app/domain/day/entities.py`)

### `DependencyHealth`

Статус доступности внешней/локальной зависимости при старте дня.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `name` | `str` | Критично | Имя зависимости (`llm`, `qdrant`, `local_env` и т.д.). |
| `status` | `Literal["ok", "degraded", "failed"]` | Критично | Итог проверки для решений по fallback. |
| `reason` | `str \| None` | Желательно | Человеко-понятная причина деградации/ошибки. |
| `latency_ms` | `int \| None` | Опционально | Базовая метрика производительности и диагностики. |

### `StepExecutionReport`

Единый отчет выполнения любого шага оркестрации.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `step_name` | `str` | Критично | Название шага для читаемости pipeline и журналирования. |
| `status` | `Literal["success", "failed", "skipped"]` | Критично | Фактический исход шага. |
| `duration_ms` | `int` | Желательно | Длительность шага для наблюдаемости и поиска узких мест. |
| `error_message` | `str \| None` | Желательно | Текст ошибки для понятного действия пользователя. |
| `retry_count` | `int` | Опционально | Количество повторов (актуально для сценариев с retry). |

### `MorningBrief`

Утренний бриф, который получает пользователь после `start_day`.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `summary` | `str` | Критично | Короткая сводка состояния старта дня. |
| `today_focus` | `list[str]` | Критично | 1-3 ключевых фокуса дня. |
| `restored_checkpoint` | `str \| None` | Желательно | Последняя точка, откуда продолжается работа. |
| `warnings` | `list[str]` | Желательно | Предупреждения о деградации и ограничениях режима. |

### `EveningSummary`

Вечерняя сводка по итогам `end_day`.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `summary` | `str` | Критично | Короткое текстовое завершение дня. |
| `done_items` | `list[str]` | Критично | Что реально завершено за день. |
| `blocked_items` | `list[str]` | Желательно | Что заблокировано и требует внимания позже. |
| `first_step_tomorrow` | `str` | Критично | Зафиксированный стартовый шаг следующего дня. |
| `warnings` | `list[str]` | Опционально | Технические или процессные предупреждения. |

## 3) Результаты use-case (`app/domain/day/results.py`)

### `StartDayResult`

Итог работы `StartDayUseCase.execute`.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `status` | `Literal["ok", "degraded", "failed"]` | Критично | Общий статус старта дня. |
| `brief` | `MorningBrief` | Критично | Сформированный утренний бриф для пользователя. |
| `dependency_health` | `list[DependencyHealth]` | Критично | Результат health-check по зависимостям. |
| `environment_reports` | `list[StepExecutionReport]` | Желательно | Детальный отчет подготовки окружения. |
| `checkpoint_loaded` | `bool` | Желательно | Факт восстановления точки продолжения. |
| `warnings` | `list[str]` | Опционально | Сводные предупреждения верхнего уровня. |

### `EndDayResult`

Итог работы `EndDayUseCase.execute`.

| Поле | Тип | Необходимость | Описание |
|---|---|---|---|
| `status` | `Literal["ok", "degraded", "failed"]` | Критично | Общий статус завершения дня. |
| `summary` | `EveningSummary` | Критично | Итоговая вечерняя сводка. |
| `checkpoint_saved` | `bool` | Критично | Подтверждение, что точка на завтра сохранена. |
| `save_report` | `StepExecutionReport \| None` | Желательно | Подробности шага сохранения checkpoint. |
| `warnings` | `list[str]` | Опционально | Сводные предупреждения верхнего уровня. |

## 4) Минимальный payload для `DayStateRepository`

Репозиторий работает с `dict`, но формат payload лучше зафиксировать заранее.

| Ключ | Тип | Необходимость | Описание |
|---|---|---|---|
| `user_id` | `str` | Критично | Привязка состояния к конкретному пользователю. |
| `day_id` | `str` | Критично | День, к которому относится запись. |
| `last_context` | `str \| None` | Желательно | Последний зафиксированный контекст работы. |
| `first_step_tomorrow` | `str` | Критично | Точка входа следующего дня. |
| `updated_at` | `str` (ISO datetime) | Желательно | Время последнего обновления для диагностики. |

---

## 5) Практический приоритет внедрения

Чтобы быстро закрыть этап 01 без перегруза кода:
1. Сначала завести только `Критично` поля во всех DTO.
2. Затем добавить `Желательно` для наблюдаемости и устойчивости.
3. `Опционально` включать по мере реальной потребности в следующих этапах.

# Этап 06. Наблюдаемость и контроль решений

## Цель этапа

Сделать работу агента прозрачной: чтобы в любой момент было понятно, что произошло, почему и что делать при отклонениях.

## Результат этапа

- Есть набор ключевых метрик по этапам работы.
- Есть удобный диагностический отчет для ежедневного использования.
- Вы можете быстро локализовать деградацию качества.

## Пошаговый план реализации

### Шаг 1. Утвердить минимальный набор метрик
- Что включить:
  - длительность шагов;
  - количество выбранных фактов;
  - частота fallback;
  - частота ошибок по компонентам;
  - доля успешных запусков start_day/end_day.
- Зачем:
  - без метрик нельзя управлять качеством системно.
- Проверка:
  - каждая метрика отвечает на конкретный вопрос.

### Шаг 2. Описать формат дневного диагностического отчета
- Что делаете:
  - определяете короткий формат:
    - здоровье инфраструктуры;
    - качество retrieval;
    - стабильность оркестрации;
    - основные инциденты.
- Зачем:
  - быстро видеть "пульс системы" без глубокого копания.
- Проверка:
  - отчет читается за 2-3 минуты.

### Шаг 3. Ввести пороги алертов
- Что делаете:
  - задаете границы "норма/риск/критично" для ключевых метрик.
- Зачем:
  - чтобы не реагировать слишком поздно.
- Проверка:
  - при выходе за порог автоматически формируется сигнал и действие.

### Шаг 4. Привязать метрики к этапам рабочего дня
- Что делаете:
  - связываете показатели с `start_day`, рабочим циклом и `end_day`.
- Зачем:
  - видеть не только "в целом плохо", а "где именно просадка".
- Проверка:
  - деградация локализуется до конкретного шага.

### Шаг 5. Сделать цикл улучшений
- Что делаете:
  - раз в день/неделю разбираете отклонения и корректируете пороги/правила.
- Зачем:
  - система растет управляемо, без хаотичных правок.
- Проверка:
  - количество повторных инцидентов снижается.

## Критерии готовности этапа

- Метрики утверждены и используются регулярно.
- Есть читаемый daily/weekly health-report.
- Реакция на инциденты стандартизирована.

## Типовые ошибки и профилактика

- Ошибка: слишком много метрик "на всякий случай".
  - Профилактика: держать только те, что реально влияют на решения.
- Ошибка: лог есть, действий нет.
  - Профилактика: у каждого сигнала должен быть playbook.
- Ошибка: нет сравнения с прошлым периодом.
  - Профилактика: отслеживать тренд, а не только текущее значение.

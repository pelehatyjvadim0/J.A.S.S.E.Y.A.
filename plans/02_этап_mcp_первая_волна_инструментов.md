# Этап 02. MCP первая волна инструментов

## Цель этапа

Подключить минимальный набор MCP-серверов, который сразу сокращает рутину и делает рабочий поток управляемым.

## Почему это второй этап

- После этапа 01 у вас уже есть сценарий дня.
- Теперь нужно дать сценарию "руки" (инструменты), чтобы он работал автоматически.

## Результат этапа

- Подключен и проверен базовый MCP-набор.
- Для каждого сервера есть четкая зона ответственности.
- Есть матрица: "команда пользователя -> какой MCP используется".

## Состав первой волны

1. Filesystem MCP
2. Terminal/Process MCP
3. Browser Automation MCP
4. Git/GitHub MCP
5. Docs/Knowledge MCP

## Пошаговый план реализации

### Шаг 1. Описать роль каждого MCP-сервера
- Что делаете:
  - фиксируете, за какие действия отвечает каждый сервер.
- Зачем:
  - чтобы избежать пересечений и неочевидного поведения.
- Проверка:
  - для каждого сервера есть 3-5 типовых задач.

### Шаг 2. Сформировать каталог команд пользователя
- Что делаете:
  - собираете список фраз-команд:
    - "запусти рабочий день";
    - "покажи где остановились";
    - "открой вкладки проекта";
    - "обнови план".
- Зачем:
  - это основа для маршрутизации интентов.
- Проверка:
  - каждая команда однозначно связывается с одним или несколькими MCP.

### Шаг 3. Спроектировать правила последовательности вызовов MCP
- Что делаете:
  - определяете, что запускается параллельно, а что строго последовательно.
- Зачем:
  - ускорение запуска + снижение вероятности гонок и конфликтов.
- Проверка:
  - для `start_day` и `end_day` есть понятный порядок вызовов.

### Шаг 4. Утвердить стратегию ошибок MCP
- Что делаете:
  - для каждого MCP описываете:
    - как выявляется ошибка;
    - как уведомляется пользователь;
    - когда выполняется fallback.
- Зачем:
  - предсказуемость и надежность рабочего процесса.
- Проверка:
  - есть не менее 1 fallback-сценария на каждый критичный MCP.

### Шаг 5. Провести smoke-проверку первой волны
- Что делаете:
  - прогоняете короткий сценарий "утро -> работа -> вечер".
- Зачем:
  - убедиться, что связка серверов реально работает как система.
- Проверка:
  - сценарий выполняется без ручных "доделок в обход".

## Критерии готовности этапа

- Есть карта ответственности MCP.
- Есть каталог пользовательских команд.
- Есть порядок выполнения и политика ошибок.
- Рабочий сценарий проходит минимум 2 раза подряд.

## Типовые ошибки и профилактика

- Ошибка: подключить много MCP сразу.
  - Профилактика: только первая волна, без голоса и сложной автоматики.
- Ошибка: нет fallback-логики.
  - Профилактика: заранее прописать "что делать при недоступности".
- Ошибка: смешать роли MCP.
  - Профилактика: один сервер — одна понятная зона ответственности.

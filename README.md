Умный русскоязычный ассистент с долговременной памятью, где каждый шаг обработки прозрачен: от поиска фактов до принятия решения, что сохранять в память.

---

## Что это за проект

**J.A.S.S.E.Y.A.** — это ассистент, который:
- общается с пользователем в CLI;
- помнит важные факты между сообщениями;
- не "магичит" в фоне, а выполняет pipeline памяти явно и предсказуемо;
- разделён на слои, чтобы легко было развивать и отлаживать код даже Junior-разработчику.

Если коротко: это практичная архитектура ассистента с управляемой памятью на `Qdrant` и `LLM`.

## Главные фишки v1.2

1. **Явный memory pipeline**  
   Память обновляется только в контролируемой точке сценария, без скрытых фоновых задач.

2. **Двухэтапный отбор фактов**  
   Сначала векторный поиск, затем rerank через LLM — это улучшает качество контекста перед ответом.

3. **Умная дедупликация памяти**  
   Для новых фактов есть развилка: `SAVE`, `JUDGE`, `SKIP`, а в пограничных случаях работает judge с решениями `REPLACE`, `ADD`, `IGNORE`.

4. **Наблюдаемость пайплайна**  
   Логи шагов и диагностические данные помогают понять, почему конкретный факт попал (или не попал) в ответ.

5. **Чистое разделение по слоям**  
   Интерфейсы, use-case, домен и инфраструктура разделены, поэтому код проще читать и менять без побочных эффектов.

## Быстрый старт

### 1) Поднять зависимости

- Запустить `Qdrant` (например через `docker-compose`).
- Убедиться, что доступна LLM по `LLM_BASE_URL` (по умолчанию `http://localhost:11434/v1`).

### 2) Проверить настройки

Все ключевые параметры лежат в `config.py`:
- модели (`BASE_MODEL`, `FACT_JUDGE_MODEL`, `MEMORY_RERANK_MODEL`);
- пороги памяти (`MEMORY_MIN_VECTOR_SCORE`, `FACT_SIMILARITY_LOW`, `FACT_SIMILARITY_HIGH`);
- размер батча (`SUMMARY_EVERY_N_MESSAGES`);
- параметры генерации (`temperature`, `top_k`, `top_p`).

### 3) Запустить ассистента

```bash
python main.py
```

## Как устроен проект

### Слои архитектуры

- `app/interfaces` — общение с пользователем (сейчас CLI-презентация ошибок и завершения).
- `app/application` — сценарии и use-case (как именно проходит один пользовательский ход).
- `app/domain` — чистые сущности и правила предметной области (без внешних вызовов).
- `app/infrastructure` — интеграции с внешним миром (LLM, embeddings, Qdrant).
- `app/core` — склейка ключевых зависимостей и общая оркестрация клиента.

### Маршрут запроса (понятно и по шагам)

1. `main.py` принимает пользовательский prompt.
2. `app/core/jasseya_client.py` собирает gateway, репозиторий памяти, reranker и use-case.
3. `app/application/chat/use_cases/handle_user_turn.py`:
   - достаёт контекст (системный prompt + выбранные факты + буфер диалога),
   - вызывает генерацию ответа,
   - передаёт ход в memory pipeline.
4. `app/application/memory/use_cases/run_memory_pipeline.py`:
   - при достижении порога формирует batch,
   - извлекает факты из пользовательских сообщений,
   - сверяет их с уже сохранёнными,
   - применяет judge, сохраняет/заменяет факты,
   - очищает обработанную часть буфера.

## Модули памяти: что за что отвечает

### `retrieve_context.py`
- Собирает сообщения для генерации ответа.
- Берёт кандидатов из памяти и ограничивает контекст полезными фактами.
- Пишет observability-метрики отбора.

### `memory_selector.py`
- Применяет reranker к кандидатам фактов.
- Возвращает итоговый список фактов и диагностику причин выбора.

### `qdrant_repository.py`
- Инициализирует коллекцию памяти.
- Сохраняет факты с метаданными.
- Ищет похожие факты с векторным score.
- Удаляет/обновляет факты при решениях judge.

### `gateway.py` + LLM-агенты
- `JasseyaAgent` — отвечает пользователю.
- `ArchivistAgent` — извлекает факты из batch.
- `JudgeAgent` — решает судьбу конфликтующих фактов.
- `LLMReranker` — уточняет релевантность найденных фактов.

## Пайплайн памяти в терминах проекта

- `SummaryBatch` — фиксированный кусок диалога на обработку.
- `summary` — список потенциально долговременных фактов.
- `reconcile` — первичная развилка `SAVE/JUDGE/SKIP`.
- `judge` — финальное решение `REPLACE/ADD/IGNORE`.

## Что важно при отладке

- Следи за шагами pipeline по порядку: `build_batch` -> `summarize` -> `reconcile` -> `persist_direct` -> `judge` -> `apply_judge` -> `trim_buffer`.
- Смотри на значения `session_id`, `trace_id`, статус шага и количество фактов на каждом этапе.
- Если факт "пропал", сначала проверяй порог векторного поиска, затем rerank-диагностику и только потом judge.

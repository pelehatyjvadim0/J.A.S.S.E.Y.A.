# Минимальный рефакторинг перед MCP

## Цель и границы

Сделать короткий "укрепляющий" рефакторинг перед внедрением MCP-ветки: убрать наиболее рискованные технические места, но не переписывать проект заново.

Что НЕ делаем в этом плане:
- не внедряем MCP;
- не меняем доменную логику памяти;
- не делаем крупный редизайн слоёв.

Что получаем в конце:
- единый и валидируемый конфиг;
- предсказуемые структурные логи;
- понятные контракты для будущих tool-вызовов;
- базовые проверки качества (типы/линт/тесты).

## Почему именно такой минимум

Текущая архитектура уже хорошая: есть разделение слоёв и use-case подход (например, [main.py](/home/user/Рабочий стол/Jasseya/main.py), [app/core/jasseya_client.py](/home/user/Рабочий стол/Jasseya/app/core/jasseya_client.py), [app/application/memory/use_cases/run_memory_pipeline.py](/home/user/Рабочий стол/Jasseya/app/application/memory/use_cases/run_memory_pipeline.py)).

Главные риски перед MCP — не в архитектурной форме, а в эксплуатационных вещах:
- конфиг сейчас статичный в [config.py](/home/user/Рабочий стол/Jasseya/config.py);
- логирование местами через `print` (в т.ч. в [app/infrastructure/llm/reranker/agent.py](/home/user/Рабочий стол/Jasseya/app/infrastructure/llm/reranker/agent.py));
- нет явного контракта будущего tool-вызова;
- нет единого "каркаса качества" (формальные проверки в CI-стиле).

## План работ

### 1) Нормализовать конфиг через Settings-слой

Файлы:
- [config.py](/home/user/Рабочий стол/Jasseya/config.py)
- новый модуль, например [app/core/settings.py](/home/user/Рабочий стол/Jasseya/app/core/settings.py)

Зачем:
- исключить расхождения значений между окружениями;
- рано ловить ошибки конфигурации (не в рантайме посреди запроса, а при старте).

Как реализовать (подпункты):
- Ввести `BaseSettings`-класс с полями текущего `config.py` (модели, порты, пороги, флаги).
- Добавить валидации: диапазоны для `temperature/top_p`, положительные порты, корректные лимиты.
- Оставить тонкий совместимый слой: текущие импорты из `config.py` временно проксировать к `settings`, чтобы не ломать проект резко.
- Добавить пример `.env.example` с понятными комментариями.

Критерий готовности:
- приложение стартует с теми же значениями по умолчанию;
- при невалидных настройках получает понятную ошибку конфигурации.

### 2) Ввести единый structured logging

Файлы:
- [app/core/logging.py](/home/user/Рабочий стол/Jasseya/app/core/logging.py)
- [app/application/memory/use_cases/run_memory_pipeline.py](/home/user/Рабочий стол/Jasseya/app/application/memory/use_cases/run_memory_pipeline.py)
- [app/application/memory/use_cases/retrieve_context.py](/home/user/Рабочий стол/Jasseya/app/application/memory/use_cases/retrieve_context.py)
- [app/infrastructure/vectorstore/qdrant_repository.py](/home/user/Рабочий стол/Jasseya/app/infrastructure/vectorstore/qdrant_repository.py)
- [app/infrastructure/embeddings/embeddings_client.py](/home/user/Рабочий стол/Jasseya/app/infrastructure/embeddings/embeddings_client.py)
- [app/infrastructure/llm/reranker/agent.py](/home/user/Рабочий стол/Jasseya/app/infrastructure/llm/reranker/agent.py)

Зачем:
- MCP-агент будет многошаговым; без нормальных логов сложно понять, где сбой и почему.

Как реализовать (подпункты):
- Добавить обёртку логгера с общими полями: `session_id`, `trace_id`, `step`, `status`, `component`.
- Заменить диагностические `print` на единый вызов логгера.
- Установить правила: пользовательские данные логировать аккуратно (без лишнего текста prompt целиком, где это не нужно).

Критерий готовности:
- у каждого шага памяти есть однотипный лог-формат;
- при ошибках легко отследить путь запроса по `trace_id`.

### 3) Подчистить типы и мелкий техдолг домена

Файлы:
- [app/domain/memory/retrieval.py](/home/user/Рабочий стол/Jasseya/app/domain/memory/retrieval.py)
- [app/domain/memory/entities.py](/home/user/Рабочий стол/Jasseya/app/domain/memory/entities.py)
- [app/application/memory/services/memory_selector.py](/home/user/Рабочий стол/Jasseya/app/application/memory/services/memory_selector.py)

Зачем:
- чистые контракты и типы — это фундамент перед добавлением tool-слоя, где ошибок из-за несоответствия структур обычно много.

Как реализовать (подпункты):
- Убрать неиспользуемые импорты и шум.
- Проверить типы полей диагностики (чтобы не было разнородных форматов в разных местах).
- Привести комментарии и docstring к единому стилю проекта.

Критерий готовности:
- типизация и читаемость лучше, без изменения поведения.

### 4) Подготовить контракты будущих tool-вызовов (без MCP-интеграции)

Файлы:
- новый модуль, например [app/domain/tools/contracts.py](/home/user/Рабочий стол/Jasseya/app/domain/tools/contracts.py)
- новый модуль, например [app/application/tools/types.py](/home/user/Рабочий стол/Jasseya/app/application/tools/types.py)

Зачем:
- при внедрении MCP не придется спорить о формате данных "на лету".

Как реализовать (подпункты):
- Ввести минимальные модели:
  - `ToolCallIntent` (имя инструмента, аргументы, причина);
  - `ToolExecutionResult` (успех/ошибка, нормализованный вывод, метаданные);
  - `AgentStepState` (шаг, цель, статус, решение о продолжении).
- Сразу добавить строгую валидацию полей и ограничений (размеры строк, типы аргументов).

Критерий готовности:
- есть стабильный контракт, который потом можно связать с MCP-адаптерами без рефакторинга домена.

### 5) Добавить каркас качества: линт + типы + покрытие

Файлы:
- конфиг инструментов качества (в зависимости от выбранного формата проекта)
- [tests/test_memory_pipeline.py](/home/user/Рабочий стол/Jasseya/tests/test_memory_pipeline.py)
- [tests/test_memory_retrieval.py](/home/user/Рабочий стол/Jasseya/tests/test_memory_retrieval.py)
- [tests/test_memory_policies.py](/home/user/Рабочий стол/Jasseya/tests/test_memory_policies.py)

Зачем:
- MCP-ветка увеличит количество edge-cases; без авто-проверок регрессии неизбежны.

Как реализовать (подпункты):
- Подключить минимальный набор: линтер, type-checker, отчёт по покрытию.
- Добавить 2-3 теста на новые settings-валидации.
- Добавить 2 теста на унифицированный формат логирования/диагностики (на уровне функций и ожидаемых полей).

Критерий готовности:
- проект проходит единый локальный quality-check командой;
- базовый набор тестов фиксирует ключевые контракты.

## Порядок выполнения (чтобы Junior не запутался)

1. Сначала `settings` (чтобы все следующие изменения опирались на стабильный конфиг).
2. Потом `logging` (чтобы видеть и дебажить остальные шаги).
3. Затем типы/чистка домена (без смены поведения).
4. Затем контракты tool-слоя.
5. В конце — тесты и quality-check.

## Риски и как их снизить

- Риск: случайно изменить runtime-поведение при переносе конфига.
  - Снижение: делать адаптер совместимости в `config.py` и прогонять текущие тесты после каждого шага.
- Риск: потерять полезную диагностику при замене `print`.
  - Снижение: заранее зафиксировать обязательные поля логов и проверять их в тестах.
- Риск: "перерефакторить" и потерять темп.
  - Снижение: ограничить объём задачами выше, всё остальное перенести в следующий этап (MCP).

## Definition of Done

- Конфиг загружается через единый settings-слой с валидацией.
- Логирование ключевых шагов унифицировано и трассируется по `trace_id`.
- Мелкий техдолг в доменных типах убран без изменения бизнес-логики.
- Контракты tool-вызовов подготовлены (но MCP ещё не подключён).
- Линт/типы/тесты запускаются и проходят на локальной машине.
